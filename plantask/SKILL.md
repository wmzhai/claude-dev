# PlanTask — 读取任务并规划实现方案

从 `tasks/` 目录中找到编号最小的待办任务文件，通过 ultrathink 深入思考分析需求，主动与用户交互确认关键细节，进入 plan mode 输出详细的实现方案，等待用户确认后再执行。

## 核心原则

- **深度思考优先**：每个关键决策点都使用 ultrathink（extended thinking）深入推理，不要急于给出方案
- **多沟通多确认**：遇到模糊、有多种实现方式、或可能影响用户体验的决策点时，主动用 `AskUserQuestion` 与用户讨论，不要自行假设
- **迭代式规划**：先理解 → 确认方向 → 深入设计 → 确认细节 → 输出最终方案

## 执行流程

### Step 1 — 定位目标任务文件

在项目根目录的 `tasks/` 下找到编号最小的 `.md` 文件（按文件名字母序排列，忽略 `done/` 子目录和 `README.md`）。

如果 `$ARGUMENTS` 不为空，则将其视为指定的任务编号或文件名（如 `T05` 或 `T05-attachment-upload.md`），定位该文件。

如果 `tasks/` 下没有待处理的任务文件，告知用户"所有任务已完成"并结束。

### Step 2 — 读取并深入理解任务

读取任务文件的完整内容，使用 ultrathink 深入思考以下内容：

- **任务标题和背景** — 这个任务要解决什么问题？为什么需要它？
- **具体需求** — 每一条功能点的深层含义和边界情况
- **涉及的关键文件** — 预判可能影响的代码区域
- **验收标准** — 每条标准的具体含义，是否有隐含的要求
- **前置依赖** — 确认是否已满足

向用户展示任务摘要，并**主动提出你在分析中发现的疑问**。

### Step 3 — 与用户确认需求细节（关键步骤）

在深入设计之前，使用 `AskUserQuestion` 与用户沟通以下方面：

1. **需求澄清**：任务描述中模糊或可能有多种理解的部分
2. **技术方向**：如果存在多种合理的实现路径，向用户说明各方案的优劣，请用户选择
3. **scope 确认**：是否有超出任务描述但用户实际期望的功能？是否有看起来在 scope 内但用户希望推迟的部分？
4. **UX 决策**：涉及用户体验的选择（交互方式、错误处理策略、数据展示形式等）

**不要跳过这一步。** 即使任务描述看起来很清晰，也至少确认一次你的理解是否正确。好的问题比好的假设更有价值。

### Step 4 — 进入 Plan Mode 深入探索

调用 `EnterPlanMode` 进入规划模式。在 plan mode 中：

1. **探索相关代码**：使用 Glob、Grep、Read、Agent 工具深入理解任务涉及的现有代码结构、模式和约定。对每个发现使用 ultrathink 思考其对实现方案的影响。
2. **分析影响范围**：确定需要创建和修改的文件清单
3. **设计实现方案**：为每个需要变更的文件编写具体的实现细节，包括：
   - 数据模型 / Schema 变更
   - Server Actions 逻辑
   - 前端组件和页面
   - 迁移和种子数据
4. **确定实现顺序**：按依赖关系排出步骤
5. **标注风险和注意事项**

**在探索过程中如果发现新的问题或关键决策点**，使用 `AskUserQuestion` 随时向用户确认，不要等到最后才暴露问题。

### Step 5 — 输出规划

规划应包含以下部分：

```markdown
# Txx: 任务标题 — 实现方案

## Context
简述当前状态和目标变更

## 与用户确认的关键决策
列出在交互中确认的重要决策，方便后续回顾

## 需要创建/修改的文件
| 文件 | 操作 | 说明 |
|------|------|------|

## 实现细节
### 1. 第一步（如 Schema 变更）
具体代码级别的描述...

### 2. 第二步
...

## 实现顺序
1. ...
2. ...

## 已知限制（v1 可接受）
- ...

## 验证方式
1. ...
```

### Step 6 — 等待用户确认

使用 `ExitPlanMode` 提交规划，等待用户审批。用户确认后方可开始编码实现。

## 注意事项

- 规划阶段**只做研究和设计**，不要编辑任何代码文件
- 充分利用 Agent 工具并行探索多个相关文件，提高效率
- 参考 CLAUDE.md 中的项目约定（Server Actions 模式、多租户隔离、认证方式等）
- 如果任务依赖的前置任务未完成，提醒用户并停止
- 规划粒度要到**具体代码变更**级别（哪个文件的哪个函数怎么改），而非泛泛而谈
- **使用 ultrathink 进行每个重要决策的推理**，确保方案经过深思熟虑
- **遇到不确定的地方一定要问用户**，宁可多问一个问题，也不要做错误的假设
