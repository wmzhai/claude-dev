# PlanTask — 读取任务并规划实现方案

从 `tasks/` 目录中找到编号最小的待办任务文件（由 `issue2task` 生成，需求和相关代码上下文已包含在内），在此基础上与用户讨论实现方案，通过 ultrathink 深入设计，进入 plan mode 输出详细的实现方案，等待用户确认后再执行。

## 核心原则

- **深度思考优先**：每个关键决策点都使用 ultrathink（extended thinking）深入推理，不要急于给出方案
- **多沟通多确认**：遇到模糊、有多种实现方式、或可能影响用户体验的决策点时，主动用 `AskUserQuestion` 与用户讨论，不要自行假设
- **迭代式规划**：先理解 → 确认方向 → 深入设计 → 确认细节 → 输出最终方案

## 执行流程

### Step 1 — 定位目标任务文件

在项目根目录的 `tasks/` 下找到编号最小的 `.md` 文件（按文件名字母序排列，忽略 `done/` 子目录）。

如果 `$ARGUMENTS` 不为空，则将其视为指定的任务编号或文件名（如 `T05` 或 `T05-attachment-upload.md`），定位该文件。

如果 `tasks/` 下没有待处理的任务文件，告知用户"所有任务已完成"并结束。

### Step 2 — 读取任务，在已有上下文基础上深入代码

读取任务文件的完整内容。**前提**：任务文件由 `issue2task` 生成，需求已澄清，且「相关代码」section 已记录了关键文件和现状分析，不需要从零开始探索。

使用 ultrathink 深入思考以下内容：

- **需求回顾** — 快速理解任务要做什么（需求已明确，不需要重新讨论）
- **验收标准** — 每条标准对应的技术实现要求
- **前置依赖** — 确认是否已满足
- **从「相关代码」出发深入探索** — 任务文件已列出关键文件和模块，从这些入口出发，进一步阅读代码细节，重点思考**如何改、怎么扩展、有什么可复用的**
- **可复用的模式** — 发现项目中已有的模式、组件、工具函数

向用户展示任务摘要和实现层面的初步思考。

### Step 3 — 与用户讨论实现方案（关键步骤）

**需求本身已在任务文件中明确，这里聚焦实现方案的讨论。** 使用 `AskUserQuestion` 与用户沟通以下方面：

1. **技术方案选型**：如果存在多种实现路径，向用户说明各方案的优劣（性能、复杂度、可维护性），请用户选择
2. **现有代码复用**：展示探索中发现的可复用模式/组件/工具函数，和用户确认是否采用，还是需要新写
3. **数据模型设计**：schema 变更方案、字段设计、关联关系等
4. **交互实现方式**：组件选择、状态管理、API 设计等具体技术决策

**不要跳过这一步。** 即使实现路径看起来很明确，也至少确认一次你的技术方案是否符合用户预期。好的技术讨论能避免返工。

### Step 4 — 进入 Plan Mode 深入设计

调用 `EnterPlanMode` 进入规划模式。在 plan mode 中：

1. **深入代码细节**：基于 Step 2 的探索结果，进一步深入需要修改的具体代码。对每个发现使用 ultrathink 思考其对实现方案的影响。
2. **分析影响范围**：确定需要创建和修改的文件清单
3. **设计实现方案**：为每个需要变更的文件编写具体的实现细节，包括：
   - 数据模型 / Schema 变更
   - Server Actions 逻辑
   - 前端组件和页面
   - 迁移和种子数据
4. **确定实现顺序**：按依赖关系排出步骤
5. **标注风险和注意事项**

**在探索过程中如果发现新的问题或关键决策点**，使用 `AskUserQuestion` 随时向用户确认，不要等到最后才暴露问题。

### Step 5 — 输出规划

规划应包含以下部分：

```markdown
# Txx: 任务标题 — 实现方案

## Context
简述当前状态和目标变更

## 技术决策记录
列出在交互中确认的实现层面关键决策，方便后续回顾

## 需要创建/修改的文件
| 文件 | 操作 | 说明 |
|------|------|------|

## 实现细节
### 1. 第一步（如 Schema 变更）
具体代码级别的描述...

### 2. 第二步
...

## 实现顺序
1. ...
2. ...

## 已知限制（v1 可接受）
- ...

## 验证方式
1. ...
```

### Step 6 — 等待用户确认

使用 `ExitPlanMode` 提交规划，等待用户审批。用户确认后方可开始编码实现。

## 注意事项

- 规划阶段**只做研究和设计**，不要编辑任何代码文件
- 充分利用 Agent 工具并行探索多个相关文件，提高效率
- 参考 CLAUDE.md 中的项目约定（Server Actions 模式、多租户隔离、认证方式等）
- 如果任务依赖的前置任务未完成，提醒用户并停止
- 规划粒度要到**具体代码变更**级别（哪个文件的哪个函数怎么改），而非泛泛而谈
- **使用 ultrathink 进行每个重要决策的推理**，确保方案经过深思熟虑
- **遇到不确定的地方一定要问用户**，宁可多问一个问题，也不要做错误的假设
