# CheckTask — 任务验收与状态更新

检查 `tasks/` 目录下编号最小的任务文件，逐项核对验收标准，更新 checkbox 状态。全部通过后，比对任务描述与实际实现的差异并同步更新任务文件，然后移入 `tasks/done/`。最后执行 `/simplify` 对本次变更的代码进行精简审查。

## 执行流程

### Step 1 — 定位目标任务文件

在项目根目录的 `tasks/` 下找到编号最小的 `.md` 文件（按文件名排序，忽略 `done/` 子目录）。

如果 `$ARGUMENTS` 不为空，则将其视为指定的任务文件名（如 `T04` 或 `T04-topic-grade-level.md`），定位该文件。

如果 `tasks/` 下没有待处理的任务文件，告知用户"所有任务已完成"并结束。

### Step 2 — 读取并解析验收标准

读取任务文件内容，找到 `## 验收标准` 部分，提取所有 checkbox 行（`- [ ]` 或 `- [x]` 格式，也兼容 `1. [ ]` 数字列表格式）。

向用户展示任务标题和验收标准列表。

### Step 3 — 逐项验证

对每一条验收标准进行验证：

- **代码相关**：检查对应的文件是否存在、相关代码是否已实现（用 Grep/Glob/Read 工具检查）
- **CI 相关**：如果有 `bun run ci` 等验收项，实际运行命令验证
- **数据/迁移相关**：检查迁移文件是否存在，schema 是否一致
- **功能相关**：检查 Server Actions、组件、路由等是否按需求实现

每项验证后输出结果（通过/未通过及原因）。

### Step 4 — 更新 checkbox 并判断结果

将已通过的验收项从 `[ ]` 更新为 `[x]`，使用 Edit 工具逐项更新。然后判断验收结果：

- **部分未通过**：列出未通过的项，保留文件在 `tasks/` 中，向用户报告哪些项需要修复，跳到 Step 7。
- **全部通过**：继续执行 Step 5。

### Step 5 — 同步任务描述与实际实现

将任务文件中的需求描述（如实现思路、技术方案、接口设计等）与代码的实际实现进行比对。开发过程中需求可能经过调整，实际实现可能与最初任务描述不一致。

- 对比任务文件中描述的方案与代码实际实现的差异
- 如果存在不一致（例如：字段名变了、方案改了、新增或删减了功能点），使用 Edit 工具更新任务文件，使其准确反映最终实现
- 只修改与实现不一致的描述部分，不改变文件整体结构
- 向用户报告做了哪些同步修改

如果描述与实现完全一致，跳过此步骤。

### Step 6 — 移入 done 目录

将任务文件移入 `tasks/done/` 目录（`git mv tasks/Txx-xxx.md tasks/done/`），向用户报告任务已完成。

### Step 7 — 执行代码精简审查

无论任务是否全部通过，最后执行 `/simplify` 对本次相关变更的代码进行精简和质量审查。

## 输出格式

```
📋 任务: T04 — Topic 学段改造

验收标准:
  ✅ 1. Schema migration 正常运行
  ✅ 2. 教师可以选择按班级或按学段发布 Topic
  ❌ 3. 某某功能未实现（原因说明）
  ...

结果: 7/8 通过
状态: 部分未通过，需要修复第 3 项
```

## 注意事项

- 不要自行修复代码问题，只做检查和报告
- `git mv` 移动文件时不要 commit（遵守 CLAUDE.md 中禁止自行提交的规定）
- 如果验收标准描述模糊无法自动验证，标记为"需人工确认"而非直接判定通过
- CI 验证只在验收标准明确要求时才运行，避免不必要的耗时
